Chapter 01
Tool Overview & Architecture
MMB Agent Sites ka poora system ek nazar mein

MMB Agent Sites Tool ek automated web-traffic generation system hai jo antidetect browser profiles ke through real articles padhta hai — Google search se leke article scroll tak — bilkul ek real insaan ki tarah. Iska main purpose website ke AdSense/ad revenue ke liye authentic-looking traffic generate karna hai.

System Architecture

Frontend
React UI — Port 5200
Vite + React + Tailwind CSS. Dashboard, Profiles, Scheduler, Analytics, Settings — sab pages yahan hain. Browser mein http://localhost:5200 pe khulta hai.
Backend
Node.js Server — Port 3200
Express.js server jo saare kaam karta hai: profile launch, CDP automation, analytics save, scheduler run, backlink build. File: MMB AGENT SITES/server/index.cjs
AI Brain
Claude Haiku (Anthropic)
Har profile ka apna AIBrain instance hota hai. Ye search query, read depth, aur next action decide karta hai. File: server/AIBrain.cjs
Automation
Playwright CDP Agent
Har browser profile ke andar Playwright connect hota hai via Chrome DevTools Protocol (CDP). File: server/agent.cjs

Data Flow — Ek Line Mein

# User → UI → Backend → Browser Provider → Profile Launch → CDP → Agent → Article Read → Analytics

Settings Save → settings.json (data folder)
Schedule Start → Multilogin/MoreLogin API → Browser opens
CDP Connect → Playwright → Agent starts reading
Article Read → analytics_data.json + read_history.json
Session Done → Browser closes → Port cache clear

Chapter 04
Profile Start — Step by Step
Jab tum "Run" dabate ho tab backend mein kya hota hai

Full Profile Start Sequence

1
User Schedule/Manual Start Triggers UI se POST /api/scheduler/run ya /api/manual/start call hoti hai. Profile IDs aur article list backend ko milti hai.
2
Launch Queue Mein Entry ProfileQueue check karta hai — kya 5 se zyada profiles already start ho rahe hain? Agar haan, to wait karta hai. Ye queue Multilogin ko overwhelm hone se bachata hai.
3
startProfileForSchedule() — 4 Attempts Attempt 1: Provider API se profile start karo (Multilogin Launcher: port 45001)
Attempt 2: CDP port cache check karo — shayad already running ho
Attempt 3: Stop karke fresh restart karo
Attempt 4: Port range 35000–65000 scan karo running browser dhundhne ke liye
4
CDP Port Milta Hai Multilogin browser khulta hai aur ek CDP debug port return karta hai (e.g. 45231). Ye port data/cdp_ports.json mein save ho jaata hai.
5
CDP Port Ready Check (pollCdpPort) Browser fully start hone mein time lagta hai. Backend 45 seconds tak har 2 sec mein http://127.0.0.1:{port}/json/version check karta hai. Ready hone par aage badhta hai.
6
ProfileAgent Create & Connect Ek naya ProfileAgent object banta hai. Profile ka persona (deterministic), session intent, bounce chance, AI Brain — sab initialize hote hain. Playwright CDP se connect hota hai.
7
Article Reading Session Agent warmup karta hai (homepage visit), phir articles padh ke analytics save karta hai. Iske baad agle chapter mein detail hai.
✅ Success State
Profile status: starting → connected → reading → done
Har step backend console mein log hota hai: [Sites-XXXX] [info] Connected! CDP:45231
❌ Failure State
Agar 4 attempts ke baad bhi CDP port nahi mila → status: error
Common reasons: Multilogin launcher not running, profile ID invalid, network issue

Chrome DevTools Protocol (CDP)

Multilogin browser khulne par ek debug port expose karta hai (e.g. 45231). Playwright usi port se connect karke browser ko bilkul ek real user ki tarah control karta hai — no extension fingerprint, no webdriver flag.

// agent.cjs — connect() method
this.browser = await chromium.connectOverCDP(
  `http://127.0.0.1:${this.debugPort}`,
  { timeout: 60000 }
);
this.context = this.browser.contexts()[0];
// 3 attempts with 6s gap between each

1
Persona Assignment (Deterministic) Profile ID se ek hash calculate hota hai. Is hash se persona pick hota hai — same profile ko hamesha same persona milta hai. 5 personas available hain.
2
Session Intent Pick research (30%) / casual (40%) / discovery (30%) — per-session thoda random. Research = deep read, Discovery = zyada articles fast.
3
Bounce & ReRead Chance Set Persona ke base value se ±5% random variation. Ye ensure karta hai ki same persona ke different profiles thode alag behave karein.
4
AIBrain Initialize ANTHROPIC_API_KEY check hoti hai. Agar set hai → AI enabled, warna → off (graceful fallback). Per-session max 25 Claude calls.
5
Warmup (Homepage Visit) First article se pehle agent site ka homepage visit karta hai — jaise real user direct URL type karne ki bajay pehle site dekhe. Researcher/DeepDiver nav menu bhi hover karte hain.

Chapter 09
Article Reading — Scroll & Dwell
3-phase reading rhythm jo real user ki tarah article padhta hai

Dwell Time Calculation

Article kab tak padha jaata hai — ye multiple factors se decide hota hai:

// Dwell time calculation order
1. User config: readTimeMin/readTimeMax (schedule se)
2. Category override: URL mein "mesothelioma" → 8–12 min, "law" → 8–12 min
3. Session intent: research → ×1.3, discovery → ×0.75
4. AI depth: Claude returns 0.2–1.0 multiplier
5. Early bounce: bounceChance % probability → only 15–45% of full time

Final: readTime × intentMult × aiDepthMult (or early bounce cutoff)

3-Phase Scroll Rhythm

Phase 1: INTRO (0–20%)
Fast Scroll
First 20% quickly scroll karo — headline aur intro dekh ke decide karo padhen ya nahi. Speed: persona.fast pixels/step.
Phase 2: BODY (20–80%)
Slow Scroll (Main Reading)
Main content yahan hota hai — ads bhi yahan show hote hain. Speed: persona.body pixels/step. Sab se zyada time yahan bitata hai.
Phase 3: OUTRO (80–100%)
Medium Scroll
Conclusion aur comments skim karo. Speed: persona.slow pixels/step. DeepDiver yahaan comments padhta hai.

Ad Detection & Pause System

Scroll karte waqt agent ads detect karta hai aur unpe pause karta hai — jaise real user ad dekhe:

  • AdSense iframes: ins.adsbygoogle, iframe[src*="ad"]
  • Site-specific ad zones: .ad-zone-in-content, .ad-zone-sidebar, div[class*="ad-zone"]
  • Ad milne par 0.5–2.5 seconds pause (configurable)
  • Ad pause count analytics mein track hota hai (adImpressions)

Read History (Duplicate Prevention)

Har padha hua article 24 hours ke liye data/read_history.json mein save hota hai. Agla session fresh articles hi assign karta hai:

// Same article dobara nahi padha jayega 24h mein
freshArticles = allArticles.filter(a => !recentUrls.has(a.url));
// recentUrls = last 24h mein padhe articles

Human Simulation Details
Woh sab techniques jo traffic ko real dikhati hain

Bezier Curve Mouse Movement

Mouse kabhi straight line mein nahi chalta — cubic bezier curve use hoti hai:

// Cubic Bezier formula — 16–28 intermediate points
bezierPt(p0, p1, p2, p3, t) {
  u = 1 - t;
  return u³×p0 + 3u²t×p1 + 3ut²×p2 + t³×p3
}
// Ease in/out timing: faster mid-move, slower at start/end
ease = 0.4 + sin(t × π) × 0.6

Keystroke Dynamics

FeatureDetail
Typing speedPersona ke hisaab se 40–300ms per character (Researcher slow, Skimmer fast)
Word ease in/outHar word ke start/end pe thoda slow, middle mein fast
Typo correction1.5% chance per character — adjacent key type hoti hai, phir Backspace
Thinking pauses4% chance per character — 250–700ms extra pause (sooch raha hai)
Word boundary pause18% chance after space — 120–500ms natural pause

Other Human-Like Behaviors

🔄 Re-Read (Scroll Back)
bounceChance se alag — reReadChance % probability se article ka kuch hissa dobara padhta hai. Scroll up → pause → scroll down thoda. DeepDiver: 75% chance, Skimmer: 8%.
🍪 Cookie Banner Dismiss
17+ CSS selectors detect karte hain cookie/newsletter popups. 600–1800ms delay ke baad dismiss hota hai (jaise real user notice kare phir close kare).
↕️ Tab Out Simulation
Casual/Skimmer/MobileUser kabhi kabhi "tab out" karte hain — scroll ruk jaati hai jaise user kisi doosre tab pe gaya. Random pause 2–8s. Phir wapas aata hai.
📱 Mobile Burst Scroll
MobileUser persona burst mode mein kaam karta hai: fast scroll (5–15 steps) → hard stop → pause 1–4s → phir burst. Fat-finger jitter: ±50% extra randomness.
🎯 Micro-Interactions
Har N scroll steps pe micro-interaction: text selection, hover on paragraph, mouse movement. N = persona.microFreq (6 = DeepDiver frequent, 20 = Skimmer rare).
🔗 Next/Related Post Click
Discovery intent ke profile: article khatam hone par related post ya next post dhundhta hai aur click karta hai — without closing the current session.

Scroll Jitter (Natural Imprecision)

Har scroll step pe ±jitter pixels add hota hai — bilkul real mouse wheel ki tarah:

scrollPx = baseScrollPx × (1 + (random - 0.5) × 2 × jitterRange)
// jitterRange: Researcher 0.35, Mobile 0.50

AI Brain — Claude Integration
Har profile ka apna Claude AI jo real-time decisions leta hai

AI Brain Kya Karta Hai?

Bina AI ke: har profile pre-defined rules follow karta hai (persona system).
AI ke saath: Claude har session mein real decisions leta hai — kya search karu, kitna padhu, kya karu aage.
Model: claude-haiku-4-5-20251001 (fastest + cheapest)  ·  Max: 80 tokens/call  ·  Limit: 25 calls/session

4 Types of AI Decisions

DecisionWhenInputOutput
Search Query Google/Bing traffic type pe Article title + topic hint Natural query string (max 7 words)
Read Depth Article khulne ke baad Title + first 200 chars of article Float 0.2–1.0 (dwell time multiplier)
Next Action Article khatam hone par Articles read count + session time + last title "continue" / "related" / "bounce"
Topic Query Title unavailable hone par Topic string Search query (max 8 words)

AI System Prompts (Har Decision Ke Liye)

// Search Query
System: "You are a {personaName} searching online. Generate a natural Google search query a real person would type. Reply with ONLY the search query. Max 7 words."

// Read Depth
System: "Decide how deeply to read an article. Reply with ONLY a decimal between 0.2 and 1.0. (0.2=quick skim, 1.0=read every word)."

// Next Action
System: "Decide your next move. Reply with ONLY one word: 'continue', 'related', or 'bounce'."

Automatic Stop (Session Complete)

1
All articles padhe ja chukegetArticlesForProfile() ne jitne fresh articles the, sab read ho gaye
2
agent.disconnect() call hota haiPlaywright browser se gracefully disconnect hota hai — no crash
3
activeAgents Map se deleteProfile agent memory se clean up hota hai
4
closeProfileForSchedule() call hota haiProvider API se profile close kiya jaata hai: Multilogin → GET /api/v1/profile/stop
5
Browser window band hoti haiMultilogin browser automatically close ho jaata hai
6
progressEntry.status = 'done'Schedule status update hota hai — UI mein "done" dikhta hai

Manual Stop (UI Stop Button)

1
UI Stop button press → POST /api/manual/stopprofileId aur browserType backend ko milte hain
2
Active agent disconnectactiveAgents.get(profileId) → disconnect() → Map se delete
3
closeProfileForSchedule()Provider API se profile close — browser band

Error Recovery Stop

Agar koi unexpected error aaye reading ke dauran:

  • Agent page close karta hai (page.close())
  • 3 retries tak try karta hai (different traffic source pe)
  • 3 retries ke baad error log hota hai
  • Profile disconnect + close hota hai (zombie browser prevent)
  • Error data/agent_errors.log mein save hota hai

Chapter 17
Troubleshooting
Common problems aur unke solutions
ProblemReasonFix
All profiles show "error" Multilogin launcher nahi chal raha, ya profile IDs galat hain (MoreLogin IDs Multilogin mein nahi chalenge) Multilogin desktop app open karo. Task Manager mein MLXDesktopApp check karo. Port 45001 listen ho raha ho.
Backend start nahi hota Port 3200 already in use, ya node_modules install nahi hue netstat -ano | findstr :3200 se PID dhundho aur kill karo. Ya npm install MMB AGENT SITES folder mein.
Frontend port 5200 error Vite already chal raha hai ya koi doosri process port le rahi hai netstat -ano | findstr :5200 → process kill karo → dobara start
AI Brain "off" dikh raha hai ANTHROPIC_API_KEY set nahi, ya Settings save nahi ki Settings page → AI Brain section → API key daalo → Save dabao → Backend restart karo
Token invalid error 30-day token expire ho gaya Multilogin account se naya automation token generate karo → .env mein update karo → backend restart
CDP connection fails Browser fully start nahi hua, ya port change ho gaya data/cdp_ports.json delete karo → dobara schedule run karo. Backend 45 seconds tak poll karta hai.
Articles read nahi ho rahe (0 reads) Sab articles already read ho chuke hain (24h cache) Naye articles add karo, ya data/read_history.json delete karo (reset history)
Settings save kar ke bhi proxy kaam nahi kar raha Purana proxy config cached hai Backend restart karo — applySettingsToEnv() startup pe fresh values set karta hai
Stop button kaam nahi kar raha CDP port cache mein nahi tha (pehle wale session mein save nahi hua) Ab fixed hai — startProfile ke baad automatically cdp_ports.json mein save hota hai
